Project Summary

We created a user-friendly, fully-responsive multiple-page website for Athletica Gym leveraging the power of Bootstrap 5. The branded header has a sticker navbar with links to Home, Products, Facilities, Nutrition, Making of and Our Story. For the first-run login, the user enters their name and phone number, and their login state is saved in localStorage. When a user log offs the program, they experience a minimal registration flow. The Home area included a selected product list that shows product badges, product pricing, and a prominent More button that shows the rest of the product list. The Facilities section included a grid view of the facility's equipment, which would adapt from mobile to desktop, and the Nutrition shown a clean layout for the calculator and related recommendations based on activity level. We defined three membership plans, (Basic, Pro, and Premium) which resulted in premium features being gated for those premium members. We added member reviews to build consumer trust and social proof. Each page also offers a compact footer composed of social icons on the left and an email signup form on the far right. Overall, accessibility, multi-device display capabilities, consistent spacing and load time, fixed common layout bugs, and clean display across devices were considered.

Project Challenges & Resolutions

Below are the key issues we faced during development and the concrete steps we took to resolve them. Our aim was to keep the notes concise and easy to scan while documenting fixes we can reuse across the project. By standardizing how we serve assets, optimize images, structure responsive layouts, and approach debugging, we improved reliability and reduced time-to-fix. These decisions also give new contributors a clear starting point, explain why certain utilities and patterns were chosen, and help prevent the same problems from resurfacing as the codebase grows.

Running from Local File Paths

During the development phase, we kept running into weird problems whenever we opened the site using the file:// protocol. Changing into a local HTTP server (like Live Server or Vite preview) solved the issue all components loaded perfectly, images used the right paths, and the development matched the behavior of a real, published website.

Image Clarity & Scaling

Some assets were low-resolution or stretched beyond their native size. We replaced weak images with higher-resolution WebP/AVIF where possible, enforced stable aspect ratios with Bootstrap’s ratio-* utilities (e.g., ratio-4x3, ratio-1x1), and applied img-fluid to prevent overflow. When needed, we used object-fit: cover to avoid distortion and added lazy loading to improve performance on mobile networks.

Responsiveness Across Devices

Early layouts didn't adapt cleanly at every breakpoint. We refactored with the container/row/col-* grid, used row-cols-* to control gallery density, and balanced spacing with utilities like g-* and py-*. We verified behavior across multiple viewport widths and devices, checking keyboard navigation and zoom to ensure a usable, accessible experience.

Debugging & Stability

Our first major problem was the login system, since it was the first JavaScript we ever wrote. We didn't know how to apply proper validation rules—like enforcing a minimum name length, requiring an 8-digit password, or showing clear error messages—so the login flow was unstable and didn't behave as expected.

Team Skills & Learning Outcomes

Front-end development

We moved beyond “it works” to “it’s accessible, maintainable, and consistent.” In HTML, we used semantic structure with clear headings, landmarks, and alt text to improve accessibility and SEO. In CSS, we kept selectors shallow, avoided specificity fights, and leaned on Bootstrap utilities for spacing, layout, and typography so components stay unified. Media stayed crisp using responsive images (img-fluid) and stable aspect ratios via Bootstrap’s ratio helpers. In JavaScript, we preferred smaller, pure functions, added null checks before DOM operations, and standardized error handling and formatting—making the codebase easier to review and extend.

Collaboration & communication

We set clear roles, held short daily stand-ups, and tracked work on a simple Kanban (Backlog → Doing → Review → Done). Feature branches and pull requests formed a reliable review loop with checklists covering functionality, UX, accessibility, and responsiveness. Descriptive commit messages and pair programming on tricky areas (navbar logic, auth flow) reduced defects and spread knowledge. Async comments kept momentum high while documenting decisions.

Responsive design with Bootstrap

We built mobile-first and layered breakpoints for smooth scaling. Image grids adapt using row-cols-3 on small screens and row-cols-lg-5 on large, with g-3/g-lg-4 for balanced spacing. Layout order stays intuitive (text left, media right) using grid columns and order-* utilities. We stabilized aspect ratios with ratio-1x1 and ratio-4x3, optimized assets (WebP/AVIF), and used lazy loading to improve performance.

Problem-solving

Our debug loop: reproduce → isolate → hypothesize → test → fix → regression check. DevTools were central—DOM inspection, network throttling, performance profiling, Lighthouse audits, and the Coverage tab for unused CSS/JS. We used console tracing and breakpoints to follow event flows, and for cross-feature issues we built minimal reproductions. Small “unit-like” checks and targeted console assertions helped prevent repeat bugs as the project grew.
