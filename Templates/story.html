Project Summary

We created a user-friendly, fully-responsive multiple-page website for Athletica Gym leveraging the power of Bootstrap 5. The branded header has a sticker navbar with links to Home, Products, Facilities, Nutrition, Making of and Our Story. For the first-run login, the user enters their name and phone number, and their login state is saved in localStorage. When a user log offs the program, they experience a minimal registration flow. The Home area included a selected product list that shows product badges, product pricing, and a prominent More button that shows the rest of the product list. The Facilities section included a grid view of the facility's equipment, which would adapt from mobile to desktop, and the Nutrition shown a clean layout for the calculator and related recommendations based on activity level. We defined three membership plans, (Basic, Pro, and Premium) which resulted in premium features being gated for those premium members. We added member reviews to build consumer trust and social proof. Each page also offers a compact footer composed of social icons on the left and an email signup form on the far right. Overall, accessibility, multi-device display capabilities, consistent spacing and load time, fixed common layout bugs, and clean display across devices were considered.

Running from Local File Paths

During the development phase, we kept running into weird problems whenever we opened the site using the file:// protocol. Changing into a local HTTP server (like Live Server or Vite preview) solved the issue all components loaded perfectly, images used the right paths, and the development matched the behavior of a real, published website.

Image Clarity & Scaling

Some assets were low-resolution or stretched beyond their native size. We replaced weak images with higher-resolution WebP/AVIF where possible, enforced stable aspect ratios with Bootstrap's ratio-* utilities. For example, ratio-4x3 and ratio-1x1. Also, we applied img-fluid to prevent overflow. When needed, we used object-fit: cover to avoid distortion; however, we didnt add lazy loading as the images were not much and adding it will be useless.

Responsiveness Across Devices

In the beginning, our layout was a bit messy. It would look normal on one screen, then when the window was shrunk, everything started moving in the wrong direction. We were using lg/md/sm, and mixing it with the grid, but still some sections stretched weird or ended up in random spots. We fixed it by actually using both the right way. First, we used the basic container/row/col setup, and then row-cols-* to control how many divs show in a row, and that solved the issue. We also fixed the spacing with simple g-* and py-* instead of using extra CSS. After that, we just kept testing on different screen sizes, zooming in and out, even checking keyboard tabbing, just to make sure the whole site worked properly.

Debugging & Stability

Our first major problem was the login system, since it was the first JavaScript we ever wrote. We didn't know how to apply proper validation rules like enforcing a minimum name length, requiring an 8-digit password, or showing clear error messages so the login flow was unstable and didn't behave as expected.

Team Skills & Learning Outcomes

Front-end development

We were no longer working for "it works," rather we worked on making something accessible to all, easy to maintain, and consistent throughout. By using HTML, we made the structure of the pages crystal clear: semantic tags, headers, and meaningful images helped a user understand the sections in one go. Most of the work in CSS was required to do with styling and color, as Bootstrap handled much of the responsiveness. We set up the interface on a very simple, unified color scheme-purple, black, white, and yellow-which was applied uniformly in buttons, backgrounds, and highlights to provide users with a clean, recognizable UI. Mainly, the work done in JavaScript is focused on the logic of a login and controlling the navbar: user authentication and access to certain pages/plans, while simultaneously blocking access for such features for unregistered users. Its aim is to ensure the site acted appropriately for each level of access, while keeping the code readable.

Collaboration & communication

We could communicate with each other very straightforwardly, and it went very smoothly. We made sure the responsibilities were assigned quite clearly: one person took a page, another took care of the JavaScript, log in, navbar, access control, while one more worked on the visuals and cleaned up some of the code, removing things we did not need. We would regularly update each other on the assigned tasks with some short and quick check-ins to let each other know when we moved things to the "done" from the "doing" section. Our last session dragged for almost five hours, which was the longest we have worked at one table on everything and eliminated inconsistencies together. It helped us limit the amount of time we have probably worked in vain and also kept the final output clean and whole.

Responsive design with Bootstrap

We used a mobile-first approach to allow responsive design and promote clean visuals with our final product. Most of the layout was done with Bootstrap's grid system. The grids for images scale nicely with the usage of the row-cols-*, and we applied img-fluid consistently so the images don't stretch in the site. The layouts are still logical - text on the left, images on the right - and the breakpoints make sure the reshaping is done right on phone, tablet, and desktop.

Problem-solving

Whenever there were issues with file://, stretched images, elements moving around, or login not working - we would recreate the bug, change one thing and test it again. DevTools was our main tool - looking at console errors, inspecting elements, and using responsive to see visual glitches. After every bug was fixed we would test the login/register flow and main pages to confirm nothing else was broken.
